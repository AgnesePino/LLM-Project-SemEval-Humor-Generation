# Reranking Pipeline - MWAHAHA Task A EN

This document explains how candidates generated by `mwahaha_task_a_en.py` are sorted and selected.

## Overview

The reranking stage does not rely on a single official automatic metric. The final choice tries to approximate the challenge criterion: human pairwise preference between jokes.

The pipeline uses three layers:

1. constraint validation;
2. numerical scoring with an LLM judge and optional humor classifiers;
3. a pairwise tournament among the best candidates.

The `refine` command adds a fourth conservative check: the new candidate replaces the existing joke only if it beats the incumbent in repeated pairwise comparisons.

## 1. Validation

Before the actual reranking step, each candidate is validated.

A candidate is penalized or excluded if it:

- is empty;
- is longer than 900 characters;
- contains tabs or newlines;
- contains boilerplate such as `Joke:`, `Here is a joke:`, or `Final joke:`;
- for `word_inclusion`, does not contain both required words verbatim;
- for `news_headline`, does not keep at least a minimal connection to the headline.

Only valid candidates move on to scoring.

Main functions:

- `validate_candidate`
- `headline_related`
- `contains_verbatim`

## 2. Numerical Score

Each valid candidate is judged by Qwen in judge mode.

The prompt asks for a score from `0` to `10`, considering:

- constraint compliance;
- brevity;
- surprise;
- specificity;
- natural English;
- humor.

The model returns a JSON object like:

```json
{"score": 7.5, "reason": "short reason"}
```

This value is stored as:

```text
judge_score
```

Main functions:

- `score_candidates`
- `build_score_prompt`
- `parse_score`

## 3. Optional Humor Classifier

If one or more `--humor-model` arguments are passed, the script also computes a humor probability:

```text
humor_score = P(humor)
```

The value is between `0` and `1`, then scaled to `0-10`.

The final score becomes:

```text
final_score = (1 - humor_weight) * judge_score + humor_weight * (10 * humor_score)
```

With the command used in the original run:

```text
--humor-weight 0.25
```

the formula was:

```text
final_score = 0.75 * judge_score + 0.25 * (10 * humor_score)
```

With the recommended `refine` command:

```text
--humor-weight 0.20
```

the formula becomes:

```text
final_score = 0.80 * judge_score + 0.20 * (10 * humor_score)
```

Example:

```text
judge_score = 8.0
humor_score = 0.75
humor_weight = 0.20

final_score = 0.80 * 8.0 + 0.20 * 7.5
final_score = 7.9
```

Therefore, the classifiers do not decide by themselves. They only provide an auxiliary signal.

## 4. Candidate Sorting

After scoring, valid candidates are sorted by:

```text
candidate.score
```

from highest to lowest.

Invalid candidates are placed at the bottom.

This is the first filtering step.

## 5. Pairwise Tournament

After numerical scoring, the script does not automatically choose the candidate with the highest score.

Instead, it takes the top `--rerank-top-k` candidates and runs a small pairwise tournament.

Example with:

```text
--rerank-top-k 4
```

The flow is:

```text
champion = candidate #1

champion vs candidate #2 -> new champion
champion vs candidate #3 -> new champion
champion vs candidate #4 -> final winner
```

At this stage, Qwen receives two jokes and must choose which one would win a human pairwise humor battle.

The prompt asks for a JSON object:

```json
{"winner": "A", "reason": "short reason"}
```

Main functions:

- `pairwise_tournament`
- `judge_pair`

## 6. Difference Between `run` And `refine`

### `run`

In the `run` command, the final joke for each ID is the winner of the pairwise tournament among the best candidates generated from scratch.

Schema:

```text
generate candidates
validate
judge score
humor classifier score
sort by final_score
pairwise tournament among top-k
write winner
```

### `refine`

In the `refine` command, there is already a current joke, called the incumbent.

The script generates new challengers only for IDs considered weak, then selects the best challenger with the same system:

```text
generate challenger candidates
validate
judge score
humor classifier score
style penalties
sort
pairwise tournament among top-k
best challenger
```

Then it compares:

```text
incumbent vs best challenger
```

The challenger replaces the incumbent only if it wins enough votes.

With:

```text
--pairwise-votes 3
--replace-votes 2
```

it needs at least:

```text
2 wins out of 3
```

This makes refinement conservative: when the result is uncertain, the already valid joke stays in place.

## 7. `refine`-Specific Penalties

During refinement, a style penalty is applied after numerical scoring.

The penalty lowers `candidate.score` if the text contains patterns that were too frequent in the submission:

- starts with `I tried`;
- starts with `I asked`;
- starts with `I told`;
- contains `turns out`;
- contains unnecessary quotation marks;
- is too long;
- contains generic phrases such as `the kind of headline`, `sounds like the news`, `my coffee`, or `press secretary`.

Conceptual example:

```text
initial score = 8.4
penalty for "I tried" = 2.0
penalty for "turns out" = 1.25

final refine score = 8.4 - 3.25 = 5.15
```

Main functions:

- `refine_style_penalty`
- `apply_refine_style_penalties`

## 8. What To Check In Diagnostics

The diagnostics JSON files contain one entry per candidate:

```json
{
  "text": "...",
  "style": "...",
  "seed": 123,
  "temperature": 0.92,
  "score": 7.9,
  "judge_score": 8.0,
  "judge_raw": "...",
  "humor_score": 0.75,
  "valid": true,
  "invalid_reason": ""
}
```

Most important fields:

- `score`: final score used to sort candidates;
- `judge_score`: Qwen score from 0 to 10;
- `humor_score`: classifier humor probability;
- `valid`: whether the candidate satisfies the constraints;
- `invalid_reason`: exclusion reason;
- `judge_raw`: raw explanation from the judge.

Refinement diagnostics also contain:

```json
{
  "incumbent": "...",
  "challenger": "...",
  "challenger_votes": 2,
  "vote_trace": ["B", "A", "B"],
  "replaced": true
}
```

Here:

- `A` = incumbent;
- `B` = challenger;
- `replaced: true` means the challenger replaced the previous joke.

## Summary

The main metric is not BLEU, ROUGE, or text similarity.

The practical metric is:

```text
estimated preference from an LLM judge in a human pairwise humor battle
```

supported by:

- constraint validation;
- local humor classifiers;
- anti-template penalties during refinement;
- conservative comparison against the incumbent.
